PostgreSQL + JSONB: por que muita gente escolhe isso em vez do MongoDB

Quando alguém precisa armazenar documentos JSON, a primeira opção que costuma vir à cabeça é o MongoDB.

Mas hoje o PostgreSQL, usando JSONB, resolve esse problema de forma tão eficiente que, em muitos projetos, não faz sentido adicionar outro banco apenas para isso.

O que é JSONB?

O JSONB é um tipo de dado do PostgreSQL que armazena documentos JSON em formato binário.

Na prática isso significa:

  • consultas rápidas em campos internos;
  • indexação dos atributos do JSON;
  • suporte completo a transações (ACID);
  • possibilidade de misturar dados relacionais e documentos na mesma tabela.

    • Exemplo:

    
"text-blue-">400 font-bold">CREATE TABLE usuarios (
    id "text-blue-">400 font-bold">SERIAL "text-blue-">400 font-bold">PRIMARY KEY,
    nome "text-blue-">400 font-bold">TEXT,
    perfil "text-blue-">400 font-bold">JSONB
);

    
{
  300 font-medium">"tema": 400">"dark",
  300 font-medium">"notificacoes": 400 font-bold">true,
  300 font-medium">"redes": {
    300 font-medium">"github": 400">"thiagolucio",
    300 font-medium">"linkedin": 400">"thiagolucio"
  }
}

    Você pode consultar qualquer campo:

    
"text-blue-">400 font-bold">SELECT *
"text-blue-">400 font-bold">FROM usuarios
"text-blue-">400 font-bold">WHERE perfil->>"text-emerald-400">'tema' = "text-emerald-400">'dark';

    Ou até indexar esse atributo para ficar extremamente rápido.

    Por que isso pode ser melhor que MongoDB?

    1. Um banco para tudo

    Em vez de manter PostgreSQL para dados relacionais e MongoDB para documentos, você usa apenas PostgreSQL.

    Menos infraestrutura, menos backup, menos monitoramento e menos complexidade.

    ---

    2. ACID de verdade

    PostgreSQL sempre teve um dos modelos de transação mais robustos do mercado.

    Quando várias operações precisam acontecer juntas, você tem garantias fortes de consistência.

    ---

    3. JOINs continuam existindo

    Se amanhã parte do JSON virar uma tabela normal, basta fazer JOIN.

    Você não precisa migrar de tecnologia.

    ---

    4. Índices eficientes

    O PostgreSQL possui índices GIN para JSONB, permitindo consultas muito rápidas em documentos grandes.

    Você consegue pesquisar por chaves, valores e estruturas internas sem precisar percorrer o documento inteiro.

    ---

    5. SQL continua sendo SQL

    Você pode combinar consultas relacionais com JSON na mesma instrução.

    Exemplo:

    
"text-blue-">400 font-bold">SELECT nome
"text-blue-">400 font-bold">FROM usuarios
"text-blue-">400 font-bold">WHERE perfil->>"text-emerald-400">'tema' = "text-emerald-400">'dark'
"text-blue-">400 font-bold">AND criado_em >= "text-blue-">400 font-bold">CURRENT_DATE - "text-blue-">400 font-bold">INTERVAL "text-emerald-400">'"text-amber-400">30 days';

    Essa flexibilidade é difícil de encontrar em bancos exclusivamente orientados a documentos.

    Então MongoDB ficou ruim?

    Não.

    MongoDB continua sendo uma excelente escolha para determinados cenários, principalmente quando praticamente todos os dados são documentos altamente flexíveis e a aplicação foi desenhada em torno desse modelo.

    Mas para muitas aplicações web, APIs e sistemas corporativos, o PostgreSQL entrega o melhor dos dois mundos: dados relacionais e documentos JSON na mesma base.

    O resultado costuma ser uma arquitetura mais simples, menos serviços para manter e uma curva de aprendizado menor para quem já trabalha com SQL.

    No fim, a melhor tecnologia depende do problema. Só vale lembrar que, em 2026, escolher MongoDB apenas porque "precisa guardar JSON" já não é uma decisão tão óbvia quanto era alguns anos atrás.
